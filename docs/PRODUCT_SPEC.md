# 타요 제품 명세서 (Codex용)

## 1. 제품 한 줄 설명

가톨릭대학교 성심교정 학생이 **같은 학교 학생끼리 택시 합승 파티를 빠르게 만들고 참여**할 수 있는 웹 MVP.

## 2. 배경과 문제

- 역에서 학교까지 거리가 길고 오르막길이 많아 도보 피로가 큼
- 시간에 쫓기면 택시 수요가 생기지만 혼자 타기엔 비용 부담이 큼
- 기존 학교 커뮤니티의 즉석 모집 글은 번거롭고 신뢰성이 낮음
- 따라서 **학교 인증 기반의 즉시 택시 합승 매칭 서비스**가 필요함

## 3. 대상 사용자

- 1차 대상: 가톨릭대학교 성심교정 학생
- 확장 전제: 향후 다른 대학으로 확대 가능
- 가입 제한: `@catholic.ac.kr` 이메일 계정 소유자만

## 4. MVP 목표

### 기능 목표
- 학교 이메일 인증 후 로그인 가능
- 택시팟 생성 가능
- 택시팟 목록 조회 및 필터 가능
- 택시팟 참여 / 취소 가능
- 탑승 후 평가 및 신고 가능
- 마이페이지에서 내 정보와 이용 히스토리 확인 가능

### 서비스 목표
- 학생 간 택시 합승을 빠르게 성사시키는 이동 매칭 서비스

## 5. 비범위(이번 버전에서 제외)

- 실시간 채팅
- 택시 자동 호출
- 자동 결제 / 비용 정산
- 푸시 알림
- 다학교 멀티테넌시
- 정교한 신고 처리 워크플로 자동화
- 모바일 앱
- 관리자용 복잡한 통계 대시보드

## 6. 사용자 역할

### 일반 사용자
- 학교 이메일로 로그인
- 택시팟 생성
- 택시팟 검색/필터
- 참여/취소
- 후기 작성
- 신고 접수
- 마이페이지 조회

### 관리자
- 일반 사용자 화면에서는 관리자 가입 진입점 없음
- DB 또는 시드 데이터로만 관리자 권한 부여
- 전체 사용자 / 택시팟 / 신고 조회 가능

## 7. 정보 구조 / 주요 라우트

- `/` : 랜딩 또는 로그인 여부에 따른 리다이렉트
- `/login` : 학교 이메일 OTP 로그인
- `/home` : 요약 화면(다가오는 내 모임, 추천/최근 택시팟)
- `/parties` : 택시팟 목록 페이지
- `/parties/new` : 택시팟 생성 페이지
- `/parties/[id]` : 택시팟 상세 페이지
- `/feedback/[partyId]` : 피드백/신고 페이지
- `/mypage` : 내 정보 / 히스토리
- `/admin` : 관리자 전용 페이지

## 8. 핵심 사용자 흐름

### 8-1. 회원가입/로그인
1. 사용자가 로그인 페이지 접속
2. `@catholic.ac.kr` 이메일 입력
3. 이메일 OTP 또는 매직 링크 인증
4. 최초 로그인 시 프로필 생성
5. 홈 또는 택시팟 목록으로 이동

### 8-2. 택시팟 생성
1. 사용자가 `+` 버튼 또는 생성 버튼 클릭
2. 출발지 선택
   - 카카오맵에서 검색/선택 가능
   - 또는 수동 텍스트 입력 가능
3. 출발 상세 정보 입력
   - 예: 부천역 3번 출구
4. 도착지 입력
   - 기본값: 가톨릭대학교 성심교정
   - 필요 시 수정 가능
5. 출발 시간 입력
6. 최대 인원 입력 (2~4명)
7. 메모 입력(선택)
8. 생성 완료 시 작성자는 자동으로 해당 팟의 첫 번째 멤버가 됨

### 8-3. 택시팟 참여
1. 사용자가 목록에서 원하는 팟을 클릭
2. 상세 정보 확인
3. 참여 버튼 클릭
4. 아래 조건 모두 만족 시 참여 성공
   - 같은 학교 사용자
   - 정원 미달
   - 이미 시작되지 않음
   - 사용자가 다른 활성 팟에 속해 있지 않음

### 8-4. 피드백/신고
1. 모임 시간이 지나고 1시간이 지난 뒤
2. 사용자가 앱에 재방문하거나 관련 페이지에 진입하면
3. 피드백 필요 모달/배너를 강하게 노출
4. 각 참여자에 대해 후기/신고 입력 가능
5. 제출 후 히스토리에 반영

> 주의: 웹 앱은 사용자가 앱을 보고 있지 않을 때 진짜로 “강제 페이지 전환”을 보장하기 어렵다.  
> 따라서 MVP에서는 **다음 방문 시 강제 모달/배너 + 피드백 페이지 유도** 방식으로 구현한다.

## 9. 화면 요구사항

### 9-1. 택시팟 목록 페이지(`/parties`)
- 상단: 학교명, 필터, 정렬
- 본문: 택시팟 카드 리스트
- 카드 정보
  - 출발지명
  - 출발 상세(출구 등)
  - 도착지명
  - 출발 시간
  - 현재 인원 / 최대 인원
  - 모집 상태
- 오른쪽 하단 플로팅 `+` 버튼
- 필터
  - 지역(출발지 텍스트)
  - 시간대
  - 모집 가능 여부

### 9-2. 택시팟 상세 페이지(`/parties/[id]`)
- 모임 핵심 정보 크게 노출
- 참여자 목록
- 작성자 여부 표시
- 참여 / 취소 버튼
- 이미 다른 활성 팟에 속한 경우 경고 문구 표시
- 시작 시각이 지났거나 정원이 찼으면 참여 비활성화

### 9-3. 생성 페이지(`/parties/new`)
- Kakao 지도 검색/선택 UI
- 지도 실패 시 수동 입력 UI 자동 대체
- 출발지/도착지/시간/정원/메모 입력
- 유효성 검사
- 생성 완료 후 상세 페이지 이동

### 9-4. 피드백 페이지(`/feedback/[partyId]`)
- 함께 탑승한 사용자 목록
- 각 사용자마다
  - 시간 준수 평가
  - 불편 사항 여부
  - 신고 버튼
- 전체 후기 텍스트 입력란
- 제출 후 완료 상태 표시

### 9-5. 마이페이지(`/mypage`)
- 기본 프로필
  - 이메일
  - 닉네임
  - 학교
- 히스토리
  - 참여한/생성한 과거 택시팟 목록
- 정보 수정
- 탈퇴 버튼

### 9-6. 관리자 페이지(`/admin`)
- 사용자 목록 조회
- 택시팟 목록 조회
- 신고 목록 조회
- 관리자 전용 접근 제어

## 10. 비즈니스 규칙

1. 학교 이메일(`@catholic.ac.kr`) 외 가입 불가
2. 사용자는 동시에 하나의 활성 택시팟에만 참여 가능
3. 택시팟 최대 정원은 4명
4. 생성자는 생성과 동시에 멤버로 참여 처리
5. 시작 시간이 지난 모집은 참여 불가
6. 정원이 가득 차면 참여 불가
7. 히스토리는 별도 저장 테이블 없이 과거 참여 데이터로 파생 가능
8. 신고는 관리자 페이지에서 조회 가능해야 함
9. 관리자 계정은 UI에서 생성하지 않음
10. 지도 기능 실패 시 수동 위치 입력으로 우회 가능해야 함

## 11. 기술 결정

- 프레임워크: Next.js App Router
- 언어: TypeScript (strict)
- 스타일: Tailwind CSS
- 인증/DB: Supabase
- 지도: Kakao Maps JavaScript SDK
- 장소 검색: Kakao Maps services 라이브러리
- 배포: Vercel
- 폼 검증: Zod 권장
- 상태 관리: 서버 상태는 Supabase + React Query 또는 서버 액션 중 하나로 일관되게 선택

## 12. 인증 방식 제안

### 권장
- Supabase 이메일 OTP 또는 매직 링크 로그인
- 클라이언트에서 이메일 도메인 검증
- 서버/DB에서도 동일하게 도메인 검증

### 이유
- 학교 메일을 실제로 받을 수 있어야 인증 성공
- 구현 복잡도 대비 신뢰도가 적절함

## 13. 카카오맵 구현 요구사항

1. 카카오맵 키는 절대 하드코딩 금지
2. 반드시 `NEXT_PUBLIC_KAKAO_MAP_APP_KEY` 환경변수에서 읽기
3. 클라이언트에서만 SDK 로드
4. SDK 로더는 공통 유틸로 분리
5. `autoload=false`와 `libraries=services`를 포함해서 로드
6. 지도 로드 실패 시:
   - 앱이 깨지지 않아야 함
   - 수동 출발지/도착지 입력 UI를 자동으로 제공해야 함
7. 사용자가 장소를 선택하면
   - `place_name`
   - `lat`
   - `lng`
   - 상세 설명(예: 3번 출구)
   를 함께 저장 가능하도록 설계

## 15. 구현 시 반드시 해결해야 할 엣지 케이스

- 다른 활성 팟에 이미 참여한 사용자가 새 팟에 참여 시도
- 정원 초과 참여
- 시작 시간이 지난 뒤 참여 시도
- 본인이 만든 팟에 중복 참여 시도
- 피드백 이미 제출한 사용자의 중복 제출
- 지도 SDK 미로드 / 키 누락 / 도메인 미등록
- 로그인 전 보호 페이지 접근
- 관리자 아닌 사용자의 `/admin` 접근
- 탈퇴한 사용자 처리
- 과거 팟 / 미래 팟 구분

## 16. UI/UX 원칙

- 모바일 우선 반응형
- 한 화면에서 핵심 결정이 가능해야 함
- 택시팟 카드는 한눈에 시간/장소/인원 확인 가능
- CTA는 명확하게 1개 중심
- 빈 상태/에러 상태/로딩 상태를 모두 제공
- 지도는 보조 수단, 핵심 서비스는 결국 “빠른 매칭”임
